## 6. Coding Rules

These rules apply to all four projects without exception. Any AI agent or developer working on these projects must adhere to every rule listed here.

### General JavaScript Rules

```javascript
// ✅ Always use const for values that won't be reassigned
const apiKey = import.meta.env.VITE_API_KEY;

// ✅ Use let only when reassignment is necessary
let retryCount = 0;

// ❌ Never use var
var oldStyle = 'never'; // Forbidden

// ✅ Use arrow functions for callbacks
const fetchWeather = async (city) => { ... };

// ✅ Use template literals for string interpolation
const url = `https://api.openweathermap.org/data/2.5/weather?q=${city}&appid=${apiKey}`;

// ❌ Never concatenate strings with +
const url2 = 'https://api.openweathermap.org' + '?q=' + city; // Forbidden
```

### Naming Conventions

| Entity               | Convention     | Example                          |
|----------------------|----------------|----------------------------------|
| React Components     | PascalCase     | `WeatherCard`, `TripList`        |
| Functions / Handlers | camelCase      | `handleSearch`, `fetchForecast`  |
| Variables            | camelCase      | `weatherData`, `currentCity`     |
| Constants (global)   | UPPER_SNAKE    | `MAX_SEARCH_HISTORY`, `BASE_URL` |
| CSS class names      | kebab-case     | `weather-card`, `trip-header`    |
| CSS Module classes   | camelCase      | `styles.weatherCard`             |
| Files (components)   | PascalCase     | `WeatherCard.jsx`                |
| Files (hooks)        | camelCase      | `useWeather.js`                  |
| Files (utilities)    | camelCase      | `formatDate.js`                  |
| Files (CSS Modules)  | PascalCase.module.css | `WeatherCard.module.css`  |

### Component Rules

```jsx
// ✅ All components must be functional (no class components)
const WeatherCard = ({ city, temperature, description }) => {
  return (
    <div className={styles.card}>
      <h2>{city}</h2>
      <p>{temperature}°C</p>
      <p>{description}</p>
    </div>
  );
};

export default WeatherCard;

// ❌ Class components are forbidden
class WeatherCard extends React.Component { } // Forbidden

// ✅ Destructure props in the function signature
const TripCard = ({ trip, onDelete, onEdit }) => { ... };

// ❌ Do not access props.x inside the body if destructuring is possible
const TripCard = (props) => {
  const title = props.trip.title; // Avoid — destructure instead
};
```

### Hooks Rules

```javascript
// ✅ Hooks must only be called at the top level of a component
const WeatherApp = () => {
  const [city, setCity] = useState('');
  const [weather, setWeather] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  // ...
};

// ❌ Never call hooks inside conditions, loops, or nested functions
if (condition) {
  const [state, setState] = useState(); // Forbidden
}

// ✅ Custom hooks must be named with the "use" prefix
const useWeather = (city) => { ... };
const useLocalStorage = (key, defaultValue) => { ... };
```

### Async / API Rules

```javascript
// ✅ Always use async/await with try/catch for API calls
const fetchWeatherData = async (city) => {
  setLoading(true);
  setError(null);
  try {
    const response = await axios.get(`${BASE_URL}/weather`, {
      params: { q: city, appid: apiKey, units: 'metric' }
    });
    setWeatherData(response.data);
  } catch (err) {
    if (err.response?.status === 404) {
      setError('City not found. Please check the city name.');
    } else {
      setError('Unable to fetch weather data. Please try again.');
    }
  } finally {
    setLoading(false);
  }
};

// ❌ Never use .then()/.catch() chains when async/await is available
axios.get(url).then(res => ...).catch(err => ...); // Discouraged
```

### Import Order Convention

```javascript
// 1. React and React-related imports
import React, { useState, useEffect } from 'react';
import { useNavigate } from 'react-router-dom';

// 2. Third-party library imports
import axios from 'axios';

// 3. Internal components

```

## 7. Architecture Rules

### Component Design Principles

1. **Single Responsibility:** Each component does exactly one thing. A `WeatherCard` displays weather; it does not fetch it.
2. **Container vs. Presentational:** Keep data-fetching logic in container components or hooks; keep UI logic in presentational components.
3. **Props Down, Events Up:** Data flows downward via props; user interactions flow upward via callback functions.
4. **Avoid Prop Drilling Beyond 2 Levels:** If props need to pass through more than two layers, extract to a custom hook or lift state appropriately.

### State Management Principles

| Data Type                       | Where to Store                          |
|---------------------------------|-----------------------------------------|
| UI state (open/closed, tab)     | Local `useState` in the component       |
| Shared state (current city)     | Lifted to closest common parent         |
| Persisted data (history, trips) | `localStorage` via custom `useLocalStorage` hook |
| Server data (API response)      | Local `useState` in a container or hook |
| Complex state (fitness logs)    | `useReducer` for multi-field updates    |
| Global theme (dark mode)        | `useState` at App level, passed via props or Context |

### File Architecture Principles

- No component file should exceed 200 lines of JSX/JS
- If a component exceeds 150 lines, evaluate whether it needs to be split
- Business logic (calculations, formatting) must live in `utils/`, not in components
- API calls must live in `services/` or custom hooks, not directly in component bodies
- Constants must live in `constants/` or a `constants.js` file at the root of `src/`

### Branch Strategy

| Branch Name     | Purpose                                                        | Protected? |
|-----------------|----------------------------------------------------------------|------------|
| `main`          | Stable, production-ready code only                             | Yes        |
| `dev`           | Active development branch; features are merged here first      | No         |
| `feature/<name>`| Individual feature development (e.g. `feature/city-search`)   | No         |
| `fix/<name>`    | Bug fix branches (e.g. `fix/forecast-api-error`)               | No         |

- Never commit directly to `main` (except the initial project setup commit)
- All development work happens on `dev` or `feature/*` branches
- `dev` is merged into `main` only when a complete, tested version is ready
- Feature branches are created from `dev` and merged back into `dev`
- Delete feature branches after they are merged

### Pull Request Rules

- Merge `dev` → `main` using a PR with a descriptive title
- PR title format: `[Task N] <Summary of what this PR delivers>`
- Example: `[Task 1] Complete React Weather Hub with forecast and search history`

## 9. Commit Rules

### Commit Message Format

All commits must follow the Conventional Commits specification:

```
<type>(<scope>): <short description>

[optional body]

[optional footer]
```

### Commit Types

| Type       | When to Use                                                    |
|------------|----------------------------------------------------------------|
| `feat`     | Adding a new feature or component                              |
| `fix`      | Fixing a bug or incorrect behavior                             |
| `style`    | CSS changes, formatting — no logic changes                     |
| `refactor` | Code restructuring without adding features or fixing bugs      |
| `docs`     | README updates, comments, documentation                        |
| `test`     | Adding or updating tests                                       |
| `chore`    | Build config, dependency updates, `.gitignore`, tooling        |
| `perf`     | Performance improvements                                       |
| `ci`       | CI/CD configuration changes                                    |

### Commit Examples

```bash
# Good commit messages
git commit -m "feat(search): add city search with Enter key support"
git commit -m "fix(api): handle 404 error when city is not found"
git commit -m "style(weather-card): update card layout for mobile screens"
git commit -m "refactor(hooks): extract API logic into useWeather custom hook"
git commit -m "docs(readme): add installation instructions and screenshots"
git commit -m "chore: add .env.example file with required variable keys"
git commit -m "feat(forecast): render 5-day forecast with daily high/low"
git commit -m "fix(storage): prevent crash when localStorage is unavailable"

# Bad commit messages (forbidden)
git commit -m "update"           # Too vague
git commit -m "fixed stuff"      # Meaningless
git commit -m "WIP"              # Should not be committed to dev or main
git commit -m "asdf"             # Gibberish
git commit -m "working now"      # No indication of what changed
```

### Commit Frequency Guidelines

- Commit after completing each meaningful unit of work (not after every save)
- A minimum of 10 commits per project is expected across the project lifecycle
- Do not create one giant commit that includes everything
- Each commit should represent a logical, reviewable unit of work

### How to Generate Custom Hooks

```javascript
// src/hooks/useLocalStorage.js
import { useState, useEffect } from 'react';

const useLocalStorage = (key, initialValue) => {
  const [storedValue, setStoredValue] = useState(() => {
    try {
      const item = window.localStorage.getItem(key);
      return item ? JSON.parse(item) : initialValue;
    } catch (error) {
      console.error(`useLocalStorage: error reading key "${key}":`, error);
      return initialValue;
    }
  });

  const setValue = (value) => {
    try {
      const valueToStore = value instanceof Function ? value(storedValue) : value;
      setStoredValue(valueToStore);
      window.localStorage.setItem(key, JSON.stringify(valueToStore));
    } catch (error) {
      console.error(`useLocalStorage: error writing key "${key}":`, error);
    }
  };

  return [storedValue, setValue];
};

export default useLocalStorage;
```

## 15. Best Practices

### Performance Best Practices

| Practice                          | Implementation                                                    |
|-----------------------------------|-------------------------------------------------------------------|
| Avoid unnecessary re-renders      | Use `React.memo` for expensive pure components                    |
| Memoize expensive calculations    | Use `useMemo` for derived data                                    |
| Stable callback references        | Use `useCallback` for handlers passed to child components         |
| Lazy load routes                  | Use `React.lazy` and `Suspense` for page components in routed apps|
| Debounce search inputs            | Wait 300–500ms after keystroke before triggering API call         |
| Avoid fetching on every render    | Use `useEffect` with correct dependency arrays                    |

### Accessibility Best Practices

| Practice                          | Implementation                                                    |
|-----------------------------------|-------------------------------------------------------------------|
| Semantic HTML                     | Use `<button>`, `<nav>`, `<main>`, `<section>`, `<article>` appropriately |
| Alt text on images                | Every `<img>` must have a meaningful `alt` attribute              |
| Keyboard navigation               | All interactive elements reachable and operable via keyboard      |
| Focus indicators                  | Never use `outline: none` without providing a custom focus style  |
| Form labels                       | Every `<input>` must have an associated `<label>`                 |
| ARIA roles                        | Add `aria-label` for icon-only buttons                            |
| Color contrast                    | Text must meet at least 4.5:1 contrast ratio against background   |

### useEffect Guidelines

```javascript
// ✅ Always specify the dependency array
useEffect(() => {
  if (city) fetchWeather(city);
}, [city]); // Runs only when city changes

// ✅ Cleanup subscriptions and timers
useEffect(() => {
  const timer = setTimeout(() => fetchWeather(searchTerm), 400);
  return () => clearTimeout(timer); // Cleanup
}, [searchTerm]);

// ❌ Missing dependency array causes infinite loop
useEffect(() => {
  fetchWeather(city); // Runs on every render — dangerous
});

// ❌ Empty array when dependencies exist
useEffect(() => {
  setData(processData(rawData)); // rawData is a dependency — must include it
}, []); // Bug: rawData missing from deps
```

// Define action types as constants
const WORKOUT_ACTIONS = {
  ADD_WORKOUT: 'ADD_WORKOUT',
  REMOVE_WORKOUT: 'REMOVE_WORKOUT',
  UPDATE_WORKOUT: 'UPDATE_WORKOUT',
};

// Reducer must be a pure function
const workoutReducer = (state, action) => {
  switch (action.type) {
    case WORKOUT_ACTIONS.ADD_WORKOUT:
      return {
        ...state,
        workouts: [...state.workouts, action.payload],

    default:
      return state;
  }
};

## 22. Environment Variables Policy

### Core Rules

1. **All API keys and secrets must be stored in `.env`** — never in source code
2. **`.env` must be in `.gitignore`** — it must never be committed to Git
3. **`.env.example` must be committed** — it serves as documentation for required keys
4. **All Vite environment variables must be prefixed with `VITE_`** — this is required for Vite to expose them to the browser

### Variable Naming Convention

```
VITE_<SERVICE>_<KEY_TYPE>
```

| Variable Name                  | Project | Description                        |
|--------------------------------|---------|------------------------------------|
| `VITE_OPENWEATHER_API_KEY`     | Task 1  | OpenWeather API key                |

### `.env` File (Local — Never Committed)

```env
# .env
# ⚠️  This file contains secrets. NEVER commit this file.

VITE_OPENWEATHER_API_KEY=your_real_api_key_here
```

### `.env.example` File (Committed — Safe for Public)

```env
# .env.example
# Copy this file to .env and fill in the values.
# Never put real API keys in this file.

VITE_OPENWEATHER_API_KEY=your_openweather_api_key_here
```

### Accessing Variables in Code

```javascript
// In any JavaScript/JSX file within a Vite project
const apiKey = import.meta.env.VITE_OPENWEATHER_API_KEY;

// Validate key exists at startup (in main.jsx or App.jsx)
if (!import.meta.env.VITE_OPENWEATHER_API_KEY) {
  console.error('VITE_OPENWEATHER_API_KEY is not set. Check your .env file.');
}
```

### Vercel Environment Variables

For deployed applications, add environment variables in Vercel:

1. Go to your project on vercel.com
2. Navigate to **Settings → Environment Variables**
3. Add each variable from `.env.example` with the real values
4. Redeploy for changes to take effect

> Variables must be added to all relevant environments: Production, Preview, and Development.

## 24. Troubleshooting Guide

### Problem: `npm run dev` fails with port already in use

```bash
# Find and kill the process using port 5173
lsof -ti:5173 | xargs kill -9    # macOS / Linux
# Or use a different port
npm run dev -- --port 3000
```

---

### Problem: API key not working in development

```
Error: Request failed with status code 401
```

**Steps to resolve:**
1. Verify the `.env` file is in the project root (same level as `package.json`)
2. Verify the variable name starts with `VITE_`
3. Verify the key has been activated on the OpenWeather dashboard (can take up to 2 hours for new keys)
4. Restart the Vite dev server after editing `.env`

```bash
# Stop dev server (Ctrl+C), then restart
npm run dev
```

---

### Problem: 3D model not loading in Task 2

**Common causes:**
- Model file not in `public/` directory (use `public/models/`)
- Wrong path format — use `/models/model.glb` (absolute from public), not `./models/model.glb`
- File format not supported — use `.glb` (binary GLTF)

```jsx
// ✅ Correct path for files in public/models/
useGLTF('/models/product.glb')

// ❌ Wrong — relative path does not work in production
useGLTF('./models/product.glb')
```

---

### Problem: LocalStorage not persisting after page reload

```javascript
// Check that JSON.stringify is used when writing
localStorage.setItem('workouts', JSON.stringify(workouts));

// Check that JSON.parse is used when reading
const saved = JSON.parse(localStorage.getItem('workouts') || '[]');

// Wrap in try/catch — localStorage can throw in private browsing
try {
  const saved = JSON.parse(localStorage.getItem('workouts')) ?? [];
  setWorkouts(saved);
} catch (e) {
  setWorkouts([]);
}
```

---

### Problem: React Router shows 404 on page refresh in production

This is a Single Page Application (SPA) routing issue. The hosting platform needs to redirect all routes to `index.html`.

**Fix for Vercel:** Create `vercel.json` in the project root:

```json
{
  "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
}
```

**Fix for Netlify:** Create `public/_redirects`:

```
/*    /index.html    200
```

---

### Problem: Chart.js charts not rendering

```javascript
// ✅ Always register Chart.js components before use
import {
  Chart as ChartJS,
  CategoryScale,
  LinearScale,
  BarElement,
  Title,
  Tooltip,
  Legend,
} from 'chart.js';

ChartJS.register(
  CategoryScale,
  LinearScale,
  BarElement,
  Title,
  Tooltip,
  Legend
);
```

---

### Problem: Build fails with "cannot find module" error

```bash
# Verify all imports use the correct path (case-sensitive on Linux)
# ❌ Wrong casing
import WeatherCard from '../components/weathercard/WeatherCard';
# ✅ Correct casing
import WeatherCard from '../components/WeatherCard/WeatherCard';

# Verify the package is installed
npm install <package-name>

# Clear cache and reinstall if persistent
rm -rf node_modules package-lock.json
npm install
```

A: This internship uses JavaScript. TypeScript is not required and should not be used unless explicitly discussed with the CODTECH supervisor.
